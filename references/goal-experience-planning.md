# Goal Experience Planning

Use this when creating, reviewing, or updating a product plan.

## Core rule

A plan should describe the user's target experience, not only the feature list.

Before turning a plan into issues, identify:

- target user and situation
- user's current pain, limitation, or unmet desire
- the moment where the product clearly helps
- what the user can do, understand, decide, feel, or trust afterward
- the first session or first successful workflow
- the non-goals that protect that experience from dilution

## Goal experience shape

Capture the target experience as a compact before-and-after:

- Before: what is hard, confusing, slow, risky, or missing for the user
- During: the primary product moment, action, scene, or workflow
- After: what changed for the user in capability, confidence, clarity, time, or emotion
- Proof: what behavior, screen, copy, data, or test would show that experience exists

This applies to operational tools too. A SaaS or admin product may target speed, confidence, reduced ambiguity, fewer handoffs, or safer repeated work instead of delight or visual richness.

## Planning questions

Ask these when the plan is missing target experience:

- Who is the user in the moment this product matters?
- What should they be able to do or understand after the product helps?
- What is the first moment where they should feel the product is working?
- What must the experience avoid, such as dashboard clutter, confusing copy, too many steps, or hidden state?
- What would a user-visible proof look like?

Do not ask all questions mechanically. Pick the one or two that change the first issue wave or plan anchor.

## Issue shaping

Every early `core` issue should advance at least one part of the target experience.

If an issue cannot name the user moment it improves, classify it as `support`, `internal`, or `deviation`. Support issues should name the later core slice that makes the target experience visible.

## Drift signal

Treat the project as drifting when:

- the plan lists many features but no target user moment
- early waves produce infrastructure but no recognizable user before-and-after
- UI work looks correct but does not create the promised feeling, clarity, or trust
- copy, visuals, and proof do not support the stated user experience

Corrective waves should restore the target experience before adding more internal layers.
