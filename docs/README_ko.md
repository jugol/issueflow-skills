# Issueflow Skills

Issueflow는 Codex가 이슈 중심 개발을 안정적으로 수행하도록 돕는 스킬 팩이자 플러그인입니다. 아이디어, 버그, 제품 피드백을 바로 코드로 밀어붙이는 대신, 다음 작업을 발견하거나 명확히 하고, 이슈나 wave로 만들고, 격리된 브랜치에서 구현하고, 검증과 머지를 거쳐, 다음 에이전트를 위한 학습까지 남기게 만드는 운영 모델입니다.

핵심 철학은 단순합니다. 구현 속도가 의도, 증거, 통합 규율보다 앞서가게 두지 않는 것입니다.

## 철학

Issueflow는 소프트웨어 작업을 책임 있는 결정의 흐름으로 봅니다.

- 지금 어떤 문제를 해결하는가?
- 왜 이 문제가 중요한가?
- 무엇이 범위 안이고, 무엇이 범위 밖인가?
- 어떤 증거가 있으면 해결됐다고 말할 수 있는가?
- 어느 브랜치에서 시작해야 하는가?
- 언제 `develop`에 머지해도 되는가?
- 언제 `main`으로 릴리스해도 되는가?

AI 에이전트가 구현을 많이 맡는 환경에서는 이 질문들이 더 중요해집니다. 에이전트는 빠르지만, 빠른 만큼 범위를 넓히거나, 오래된 브랜치에서 계속 작업하거나, 채팅 안에만 남는 검증으로 마무리하기 쉽습니다.

Issueflow는 그런 흐름을 막기 위한 최소한의 구조입니다. 문서 작업을 늘리려는 것이 아니라, 나중의 사람과 에이전트가 “왜 이 변경이 있었고, 무엇으로 검증됐고, 다음 사이클은 무엇을 알아야 하는지” 이해할 수 있는 작은 기록을 남기는 것이 목적입니다.

## Two-Track Planning

Issueflow에는 두 가지 planning track이 있습니다.

Autonomous Cycle Track은 자동화, 반복 실행, post-merge 후속 작업, 실패한 테스트, 리뷰 finding, 플레이테스트나 사용자 피드백 묶음을 처리할 때 사용합니다. 에이전트는 사용자가 다음 이슈를 하나하나 말해줄 때까지 기다리지 않고, durable signal을 스캔해서 follow-up issue나 wave를 만듭니다. 독립적인 finding이 2개 이상이면 하나의 큰 이슈가 아니라 wave로 나눕니다.

Interactive Feature Intake Track은 사용자가 기능, 제품 동작, UI 변경, 넓은 개선 요청을 설명할 때 사용합니다. 에이전트는 먼저 repo context를 읽고, 중요한 질문만 짧게 묻고, tradeoff가 있는 경우 2-3개 접근안을 제안한 뒤, 승인된 방향을 이슈나 wave로 바꿉니다.

merge 또는 PR handoff 이후에는 compound loop를 실행합니다. 반복되는 문제, 재사용 가능한 패턴, 실패했던 접근, 재발 방지 규칙, follow-up trigger가 있으면 `docs/solutions/`에 남깁니다.

## 프로젝트 관리 방식

Issueflow는 Git Flow-lite 모델을 사용합니다.

- `main`: 프로덕션 또는 릴리스 기준 브랜치
- `develop`: 검증된 이슈 작업이 통합되는 브랜치
- `issue/<n>-<slug>`: 하나의 이슈만 담당하는 작업 브랜치
- `release/<version>`: 선택적인 릴리스 후보 안정화 브랜치
- `hotfix/<slug>`: `main`에서 시작하는 긴급 수정 브랜치, 이후 `develop`으로 역머지

기본 생명주기는 다음과 같습니다.

1. autonomous scan 또는 interactive brainstorm으로 작업을 발견하거나 명확히 합니다.
2. 작업을 하나의 이슈 또는 planned wave로 기록합니다.
3. `issue-intake`로 시작 가능한 상태인지 확인합니다.
4. `issue-dispatch`로 `develop`에서 이슈 브랜치나 worktree를 만듭니다.
5. 승인된 범위만 구현합니다.
6. 장기적으로 남을 테스트나 검증 경로를 추가합니다.
7. QA 증거를 만듭니다.
8. `merge-gate`를 통과합니다.
9. 통합 경로에서 충돌을 해소하며 `develop`에 머지합니다.
10. `develop` 위에서 통합 검증을 다시 실행합니다.
11. 작업 체크아웃을 `develop`에 둡니다.
12. 필요하면 `issue-compound`로 재사용 가능한 학습을 남깁니다.
13. 릴리스 준비가 되면 `release-gate`를 거쳐 `main`으로 승격합니다.

이 방식의 핵심은 “이슈 완료”와 “릴리스”를 분리하는 것입니다. 이슈 브랜치가 끝났다고 바로 `main`이 바뀌지 않습니다. 검증된 작업은 먼저 `develop`에 모이고, `main`은 의도적인 릴리스 결정으로만 바뀝니다.

증거를 통과했지만 아직 `develop`에 들어가지 않은 브랜치는 완료가 아니라 merge-ready 상태입니다. 에이전트는 그 브랜치에서 다음 이슈를 시작하면 안 됩니다.

## Vertical Slice 우선

Issueflow는 제품 작업에서 vertical slice를 우선합니다. 좋은 제품 이슈는 보통 하나의 얇은 사용자-visible 동작을 도메인, 계약, 런타임, UI, 검증 경로까지 관통해서 전달해야 합니다.

이 말은 이슈가 커야 한다는 뜻이 아닙니다. 오히려 작게 자르되, 실제 사용자 결과와 연결되어 있어야 한다는 뜻입니다.

레이어 중심 작업도 필요할 수 있습니다. 다만 그럴 때는 정직하게 분류해야 합니다.

- `core`: 핵심 사용자 경험을 직접 진전시키는 작업
- `support`: 가까운 core slice를 가능하게 하는 작업
- `internal`: 도구, QA, 관리자 화면, 리뷰 표면을 개선하는 작업
- `deviation`: 현재 계획 밖의 의도적인 탐색

`support`나 계약 우선 작업은 어떤 core slice를 가능하게 하는지 함께 적어야 합니다.

## Experience-First UI

Issueflow는 사용자-facing 제품이 QA 대시보드처럼 변하는 것을 경계합니다.

UI 작업에서 기본 화면은 사용자가 실제로 쓰게 될 제품 경험처럼 보여야 합니다. 진단 정보, raw payload, QA 라벨, 증거 패널은 테스트, 숨겨진 inspector, stable attribute, 로그, 별도 개발자 화면에 두는 것이 좋습니다.

기능 검증은 필수지만 충분조건은 아닙니다. 화면이 깨져 보이거나, 너무 일반적이거나, 시각적으로 혼란스럽거나, 사용자-facing 문구가 목표 언어의 원어민에게 어색하게 들린다면 그 이슈는 아직 끝난 것이 아닙니다.

## 지속 가능한 검증

Issueflow는 의미 있는 변경이 있을 때마다 장기 테스트 하네스가 함께 성장해야 한다고 봅니다.

예시는 다음과 같습니다.

- 버그 수정: 회귀 테스트 추가
- 기능 추가: 시나리오 테스트 추가 또는 확장
- 웹 UI: Playwright 기반 검증 추가
- Flutter UI: widget, integration, golden 테스트 추가
- 시각 변경: 렌더링 증거와 중요한 상태 보호
- 리팩터링: 기존 동작이 유지됨을 기존 또는 강화된 테스트로 증명

검증은 재실행 가능해야 합니다. 스크린샷이나 수동 확인은 보조 증거가 될 수 있지만, 자동화 가능한 동작의 유일한 증거가 되어서는 안 됩니다.

## 언제 쓰면 좋은가

Issueflow는 다음 상황에 잘 맞습니다.

- AI 에이전트가 실제 제품 구현을 많이 맡는 경우
- 여러 이슈가 병렬로 진행되는 경우
- 자동화된 반복 실행이 다음 유용한 이슈를 계속 찾아야 하는 경우
- 사용자가 설명한 기능 요구사항을 구현 전에 명확히 해야 하는 경우
- `develop`과 `main`의 브랜치 위생을 지키고 싶은 경우
- 머지 전에 재실행 가능한 검증을 남기고 싶은 경우
- 버그, 기능 요청, UX 피드백을 추적 가능한 이슈로 바꾸고 싶은 경우
- 프로젝트 계획이나 제품 문서가 실제 작업과 계속 연결되어야 하는 경우
- 미래의 에이전트가 과거 채팅 없이도 의사결정을 이해해야 하는 경우
- 웹 또는 Flutter UI 작업에 실제 브라우저/렌더링 증거가 필요한 경우

특히 AI를 적극적으로 쓰는 1인 개발자나 작은 팀에게 잘 맞습니다. 무거운 프로세스 없이도, 빠른 구현이 혼란스러운 구현으로 변하지 않게 해주는 가벼운 구조이기 때문입니다.

## 언제 과할 수 있는가

다음 상황에서는 Issueflow 전체를 적용하기엔 과할 수 있습니다.

- 유지보수하지 않을 일회성 스크립트
- 버려도 되는 아주 작은 프로토타입
- 의도적으로 trunk-based 개발만 사용하는 저장소
- 장기 증거나 미래 맥락이 중요하지 않은 변경

그래도 일부 스킬은 따로 유용할 수 있습니다. 예를 들어 `issue-raise`는 애매한 아이디어를 명확히 하는 데 좋고, `merge-gate`는 머지 전 체크리스트처럼 쓸 수 있습니다.

## 스킬들이 맡는 역할

- `issueflow`: 넓은 요청을 적절한 workflow 스킬로 라우팅합니다.
- `issue-brainstorm`: 기능 요청을 이슈로 만들기 전에 명확히 합니다.
- `repo-bootstrap`: 구체적인 저장소에 issueflow 구조를 적용합니다.
- `issue-raise`: 거친 아이디어나 피드백을 구현 가능한 이슈로 바꿉니다.
- `issue-intake`: 이슈가 시작 가능한지 판단합니다.
- `issue-dispatch`: `develop`에서 격리된 브랜치나 worktree를 만듭니다.
- `implement-web`: 승인된 웹 이슈를 구현하고 브라우저 검증을 남깁니다.
- `implement-flutter`: 승인된 Flutter 이슈를 구현하고 앱 검증을 남깁니다.
- `qa-web-proof`: Playwright와 로컬 체크로 웹 변경을 검증합니다.
- `qa-flutter-proof`: widget, integration, golden 테스트로 Flutter 변경을 검증합니다.
- `merge-gate`: 이슈 브랜치가 `develop`에 들어갈 수 있는지 판단합니다.
- `issue-compound`: 재사용 가능한 학습과 follow-up trigger를 기록합니다.
- `release-gate`: 통합된 작업이 `main`으로 승격될 수 있는지 판단합니다.

## 기대되는 결과

Issueflow를 쓰면 저장소 운영이 더 차분해져야 합니다.

- 에이전트가 어디서 시작해야 하는지 압니다.
- 브랜치는 하나의 이슈와 하나의 소유 범위를 가집니다.
- 범위 확장이 초기에 잡힙니다.
- 테스트 커버리지가 제품 범위와 함께 성장합니다.
- 릴리스가 의도적인 결정이 됩니다.
- 프로젝트 방향이 계속 보입니다.
- 이슈를 거칠수록 재사용 가능한 학습이 쌓입니다.

Issueflow는 가벼운 workflow입니다. 하지만 빠른 구현이 혼란스러운 구현이 되지 않도록 꼭 필요한 경계는 단호하게 지킵니다.
